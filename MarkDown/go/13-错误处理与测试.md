# Go 错误处理与测试

程序不可能永远不出错——文件可能不存在、网络可能断开、用户可能乱输入。Go 的哲学是：**错误也是值，要显式处理**，不要假装它不存在。这一章讲两件事：怎么优雅地处理错误，以及怎么写测试确保代码正确。

---

## 1. 错误处理基础

### 1.1 error 类型

Go 没有 try-catch。函数出错时，返回一个 `error` 类型的值。就像餐厅点菜——菜上了你要检查一下，不能端起来就吃。

```go
file, err := os.Open("不存在的文件.txt")
if err != nil {
    fmt.Println("出错了：", err) // 输出：open 不存在的文件.txt: no such file or directory
    return
}
defer file.Close()
```

**Go 的错误处理铁律**：每当函数返回 `error`，立刻检查。不要偷懒忽略它。

### 1.2 自定义错误

用 `errors.New` 创建简单的错误信息：

```go
var ErrNotFound = errors.New("用户不存在")

func findUser(id int) (string, error) {
    if id <= 0 {
        return "", ErrNotFound
    }
    return "小明", nil
}
```

用 `fmt.Errorf` 创建带格式的错误，还可以用 `%w` 包装底层错误（像套娃一样）：

```go
func readConfig(path string) error {
    _, err := os.Open(path)
    if err != nil {
        // 把底层错误包进来，同时加上自己的上下文信息
        return fmt.Errorf("读取配置文件 %s 失败: %w", path, err)
    }
    return nil
}
```

### 1.3 errors.Is 和 errors.As

`errors.Is` 判断错误是不是某个特定错误（穿透包装层）：

```go
err := readConfig("config.txt")
if errors.Is(err, os.ErrNotExist) {
    fmt.Println("配置文件不存在")
}
```

`errors.As` 把错误转换成特定类型，获取更多信息：

```go
type NotFoundError struct {
    Name string
}

func (e *NotFoundError) Error() string {
    return fmt.Sprintf("%s 未找到", e.Name)
}

var nfe *NotFoundError
if errors.As(err, &nfe) {
    fmt.Println("缺少的是：", nfe.Name)
}
```

> **要点速记**：
> - `error` 是接口，只要实现了 `Error() string` 方法就是错误
> - `errors.New()` 创建简单错误，`fmt.Errorf()` 创建格式化错误
> - `%w` 包装错误，`errors.Is` 判断错误，`errors.As` 提取错误
> - **永远不要忽略 error 返回值**

## 2. panic 和 recover

### 2.1 panic：紧急刹车

`panic` 就像火灾警报——遇到程序无法继续运行的情况才用。它会立刻中断当前函数，逐层退出，直到程序崩溃。

```go
func mustParse(s string) int {
    n, err := strconv.Atoi(s)
    if err != nil {
        panic(fmt.Sprintf("无法解析 %q 为数字", s))
    }
    return n
}
```

**什么时候该用 panic？**
- 程序初始化时缺少必要条件（如配置文件格式完全错误）
- 逻辑上不可能发生的情况（如 switch 的 default 分支）

**什么时候不该用 panic？**
- 文件不存在、网络超时这类可预期的错误 → 用 error 返回
- 用户输入了无效数据 → 用 error 返回

### 2.2 recover：灭火器

`recover` 只能在 `defer` 函数里使用，用来"接住" panic，防止程序崩溃：

```go
func safeDivide(a, b int) (result int) {
    defer func() {
        if r := recover(); r != nil {
            fmt.Println("捕获到 panic：", r)
            result = 0 // 给个默认值
        }
    }()
    return a / b // b 为 0 时会 panic
}

func main() {
    fmt.Println(safeDivide(10, 0))  // 输出：捕获到 panic：runtime error: integer divide by zero → 0
    fmt.Println(safeDivide(10, 2))  // 输出：5
}
```

工作原理：
1. `defer` 的匿名函数在 `safeDivide` 退出时执行
2. 如果发生了 panic，`recover()` 返回 panic 的值，程序继续运行
3. 如果没 panic，`recover()` 返回 `nil`，什么都不做

### 2.3 在包中使用 panic/recover 的正确姿势

库的对外 API **永远不应该暴露 panic**。内部可以用 panic，但边界上必须 recover 转 error：

```go
// 内部用 panic 简化错误处理
func doWork() {
    if somethingWrong {
        panic("内部错误")
    }
}

// 对外函数捕获 panic，转为 error 返回
func PublicAPI() (err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("内部错误: %v", r)
        }
    }()
    doWork()
    return nil
}
```

> **要点速记**：
> - `panic` 用于不可恢复的错误，`error` 用于可预期的错误
> - `recover` 只在 `defer` 函数里有效
> - 库的公共 API 不应暴露 panic，要在边界 recover 转 error
> - 日常编程 99% 的场景用 error，不用 panic

## 3. 错误处理的实用模式

### 3.1 闭包封装重复的错误处理

如果一段代码里反复做同样的错误处理（比如记日志），可以用闭包来简化：

```go
// 通用的错误处理包装器
func handleError(fn func() error) {
    if err := fn(); err != nil {
        log.Println("操作失败：", err)
    }
}

// 使用
handleError(func() error {
    return os.WriteFile("a.txt", []byte("hello"), 0644)
})
handleError(func() error {
    return os.WriteFile("b.txt", []byte("world"), 0644)
})
```

### 3.2 集中处理多个错误

Go 1.20+ 的 `errors.Join` 可以把多个错误合并成一个：

```go
err1 := doStep1()
err2 := doStep2()
err3 := doStep3()

if err := errors.Join(err1, err2, err3); err != nil {
    fmt.Println("有步骤出错了：", err)
}
```

### 3.3 启动外部命令

用 `os/exec` 包运行外部程序，也能获取它的错误：

```go
cmd := exec.Command("ls", "-la")
output, err := cmd.CombinedOutput()
if err != nil {
    fmt.Printf("命令执行失败：%v\n输出：%s\n", err, output)
    return
}
fmt.Println(string(output))
```

> **要点速记**：
> - 重复的错误处理逻辑可以抽取为闭包
> - `errors.Join` 合并多个错误
> - `os/exec` 执行外部命令，`CombinedOutput()` 获取标准输出+错误输出

## 4. 单元测试

### 4.1 测试文件和命名规则

测试文件以 `_test.go` 结尾，和被测试的文件放在同一包下：

```
calc/
├── calc.go       # 业务代码
└── calc_test.go  # 测试代码
```

测试函数必须以 `Test` 开头，参数是 `*testing.T`：

```go
// calc.go
func Add(a, b int) int {
    return a + b
}

// calc_test.go
func TestAdd(t *testing.T) {
    result := Add(2, 3)
    if result != 5 {
        t.Errorf("Add(2, 3) = %d, 期望 5", result)
    }
}
```

运行测试：`go test ./calc`

### 4.2 testing.T 的常用方法

| 方法 | 用途 |
|------|------|
| `t.Error(msg)` | 报告失败，继续执行 |
| `t.Fatal(msg)` | 报告失败，立刻停止当前测试 |
| `t.Errorf(format, args)` | 格式化报告失败 |
| `t.Logf(format, args)` | 记录日志（`go test -v` 才显示） |
| `t.Skip(msg)` | 跳过当前测试 |
| `t.Run(name, func)` | 运行子测试 |

### 4.3 表驱动测试

当测试逻辑相同但输入不同时，用"表格"组织测试数据，是最 Go 风格的写法：

```go
func TestAdd(t *testing.T) {
    tests := []struct {
        name     string
        a, b     int
        expected int
    }{
        {"正数相加", 2, 3, 5},
        {"负数相加", -1, -2, -3},
        {"零值", 0, 0, 0},
        {"正负混合", 5, -3, 2},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            result := Add(tt.a, tt.b)
            if result != tt.expected {
                t.Errorf("Add(%d, %d) = %d, 期望 %d", tt.a, tt.b, result, tt.expected)
            }
        })
    }
}
```

运行：`go test -v`，输出每个子测试的结果。

### 4.4 基准测试

基准测试以 `Benchmark` 开头，参数是 `*testing.B`，用来测量代码性能：

```go
func BenchmarkAdd(b *testing.B) {
    for i := 0; i < b.N; i++ {
        Add(2, 3)
    }
}
```

运行：`go test -bench=.`

`b.N` 是 Go 自动调整的循环次数，它会反复运行直到得到可靠的计时结果。

> **要点速记**：
> - 测试文件命名 `xxx_test.go`，测试函数命名 `TestXxx`
> - 表驱动测试是 Go 的最佳实践，用结构体切片组织测试数据
> - `t.Error` 继续跑，`t.Fatal` 立刻停
> - 基准测试用 `BenchmarkXxx`，`go test -bench=.` 运行

## 5. 性能分析入门

Go 自带性能分析工具 `pprof`，可以找到程序的"慢在哪里"。

### 5.1 在测试中生成 profile

```go
func TestWithProfile(t *testing.T) {
    // 创建 CPU profile 文件
    f, _ := os.Create("cpu.prof")
    defer f.Close()
    pprof.StartCPUProfile(f)
    defer pprof.StopCPUProfile()

    // 运行要分析的代码
    heavyWork()
}
```

### 5.2 分析 profile 文件

```bash
go tool pprof cpu.prof
```

进入交互界面后：
- `top10` 查看 CPU 消耗最多的 10 个函数
- `web` 生成可视化调用图（需要安装 graphviz）
- `list 函数名` 查看某函数的逐行耗时

### 5.3 用 go test 直接生成

```bash
# 生成 CPU profile
go test -cpuprofile=cpu.prof -bench=.

# 生成内存 profile
go test -memprofile=mem.prof -bench=.

# 分析
go tool pprof cpu.prof
```

> **要点速记**：
> - `pprof` 找性能瓶颈，`go tool pprof` 分析结果
> - `top10` 看最耗时的函数，`web` 看调用图
> - 测试时加 `-cpuprofile` / `-memprofile` 生成 profile 文件

## 新手常见坑

1. **忽略 error 返回值**：用 `_` 接收 error 后完全不处理，程序静默出错。每个 error 至少打个日志。

2. **在循环里用 panic 代替 error**：panic 是"炸弹"，会炸掉整个 goroutine。循环里应该用 error 返回，让调用者决定怎么处理。

3. **测试函数名写错**：`testAdd`（小写 t）不会被 `go test` 执行。必须是 `TestAdd`（大写 T 开头）。

4. **表驱动测试忘了 t.Run**：直接 for 循环不写 `t.Run`，测试失败时看不出是哪组数据出错。

5. **Fatal 和 Error 用混**：`t.Fatal` 会停止当前测试函数，如果后面还有必要的清理逻辑（如关闭临时文件），应该用 `t.Error` 而不是 `t.Fatal`。

6. **benchmark 里做了无关操作**：`b.N` 循环里不应该包含初始化代码，初始化要放在循环外面，否则初始化时间也会被算进性能数据。

7. **recover 不在 defer 里调用**：`recover()` 在普通函数里调用永远返回 `nil`，必须在 `defer` 的函数里才有效。
