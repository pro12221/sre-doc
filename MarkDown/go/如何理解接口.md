## 接口
接口值 = 两个指针的盒子
一个接口变量在底层长这样：
┌─────────────────┬──────────────┐
│  type pointer    │ data pointer │
│  (指向类型描述)    │ (指向实际数据) │
└─────────────────┴──────────────┘
```go
package main

import "fmt"

type Shaper interface {
	Area() float32
}

type Square struct {
	side float32
}

func (sq *Square) Area() float32 {
	return sq.side * sq.side
}

func main() {
	sq1 := new(Square)
	sq1.side = 5

	var areaIntf Shaper
	areaIntf = sq1
	// shorter,without separate declaration:
	// areaIntf := Shaper(sq1)
	// or even:
	// areaIntf := sq1
	fmt.Printf("The square has area: %f\n", areaIntf.Area())
}

```

当你写 areaIntf = sq1 时，Go 做了两件事：
1. 
检查 *Square 是否实现了 Shaper 接口（有没有 Area() float32 方法？有。✓）
2. 
装箱：把类型信息和方法表塞进第一个槽位，把 sq1 的地址塞进第二个槽位
所以赋值不是"变成"，而是包装——你把一个具体值装进了一个接口盒子。
用一个比喻
接口就像一个快递箱（上面写着"只接收有 Area 方法的物品"）。*Square 是一个实物，它碰巧有 Area 方法，所以可以被放进这个箱子。放进去之后，你只能通过箱子上的标签（Area()）来操作它，但箱子里装的始终是那个 *Square。

## 类型断言
*接口值 = (type pointer, data pointer)*。类型断言就是反向操作——从盒子里把东西掏出来。
从装箱到拆箱
areaIntf = sq1          // 装箱：*Square → Shaper
sq2 := areaIntf.(*Square)  // 拆箱：Shaper → *Square
赋值是把具体值装进接口盒子，类型断言是把接口盒子里的东西拿出来，同时告诉你"我确信里面装的是这个类型"。

赋值时：
*Square{side:5}  ──装入──▶  ┌──────────┬───────────┐
                             │ type:    │ data:     │
                             │ *Square  │ → {5}     │
                             └──────────┴───────────┘
                             这是 areaIntf (Shaper 类型)
断言时：
                             ┌──────────┬───────────┐
areaIntf.(*Square)  ◀──掏出──│ type:    │ data:     │
                             │ *Square  │ → {5}     │
                             └──────────┴───────────┘
就是看一眼 type pointer 指向的类型，如果是 *Square，就把 data pointer 返回给你；如果不是，报错。

### 两种语法，对应两种心态
心态一：我很确定
sq2 := areaIntf.(*Square)   // 如果不是 *Square，直接 panic
sq2.side = 10               // 拿出来了，随便用

心态二：我不确定，先看看
sq2, ok := areaIntf.(*Square)  // ok=true 说明确实是，ok=false 说明不是
if ok {
    sq2.side = 10
} else {
    fmt.Println("里面不是 *Square")
}

### 为什么需要断言？
因为一旦装进接口盒子，你只能调用接口定义的方法：
areaIntf.Area()      // ✓ Shaper 接口有这个方法
areaIntf.side        // ✗ 编译错误！Shaper 没有 side 字段
接口把具体类型藏起来了。你想要回 side 字段？先断言拿出来：
sq2 := areaIntf.(*Square)
sq2.side             // ✓ 拿出来了，*Square 有 side


sq1 := &Square{side: 5}
// 装箱
var areaIntf Shaper = sq1
// 此时只能用 Area()
fmt.Println(areaIntf.Area())  // 25
// 想用 side？断言拆箱
sq2 := areaIntf.(*Square)
fmt.Println(sq2.side)         // 5
// sq1、sq2、断言出来的东西，指向同一个数据
sq2.side = 100
fmt.Println(sq1.side)         // 100 —— 因为 data pointer 指向同一块内存

## 空接口
普通接口有准入条件：
type Shaper interface {
    Area() float32    // 必须有 Area 方法才能装进来
}
空接口没有准入条件：
interface{}          // 没有任何方法要求 → 任何类型都能装进来
因为准入条件是零，所以所有类型都满足空接口——这是唯一一个"无条件放行"的接口。

语法演变
// Go 1.18 之前
var x interface{} = 42
// Go 1.18 之后，alias 写法，完全等价
var x any = 42
any 就是 interface{} 的别名，仅此而已。以后建议用 any，更短更好读。

装进去容易，用起来只能当"未知物体"。想用具体功能？还是得断言拆箱：

实际用途：你需要"不知道类型"的场合

1. fmt.Println — 它不知道你会传什么
func Println(a ...any) (n int, err error)
...any 意味着你可以传任何东西进去：
fmt.Println(42)
fmt.Println("hello")
fmt.Println(&Square{side: 5})
fmt.Println(1, "two", 3.0, true)    // 全装进 []any


## 反射
接口值里藏着 (type, data) 两个指针——反射就是用代码去读这两个指针里的信息。
类型断言有一个前提：你编译期就得知道里面是什么类型。

var x any = 42
v := x.(int)       // 你得写死 int，编译时就确定了
但如果你真的不知道呢？比如你收到了一个 any，你需要知道它有几个字段、每个字段叫什么、每个字段的值是什么——你总不能把所有可能的类型都写一遍断言吧？
反射就是：在运行时动态地探查一个值的类型信息和值信息。

接口值内部的 type pointer  →  reflect.TypeOf(x)
接口值内部的 data pointer  →  reflect.ValueOf(x)


最小可运行示例
```go
package main
import (
    "fmt"
    "reflect"
)
type Square struct {
    Side float32
}
func (sq *Square) Area() float32 {
    return sq.Side * sq.Side
}
func main() {
    sq := &Square{Side: 5}
    var x any = sq        // 装箱，你已经懂了
    // 现在，在不写 *Square 的情况下，搞清楚 x 里面是什么
    t := reflect.TypeOf(x)
    v := reflect.ValueOf(x)
    fmt.Println("类型名:", t.Name())           // Square
    fmt.Println("种类:", t.Kind())             // ptr（指针）
    fmt.Println("指向的类型:", t.Elem().Name()) // Square
    fmt.Println("值:", v)                       // &{5}
    fmt.Println("能调的方法数:", v.NumMethod())    // 1
}
```
Kind —— 比 Type 更基础的概念
Type 是具体类型名（Square、int），Kind 是底层种类（struct、ptr、int）：
type MyInt int
var x MyInt = 42
reflect.TypeOf(x).Name()   // "MyInt"   —— 你起的名字
reflect.TypeOf(x).Kind()   // int       —— 底层是什么
Kind 只有这些，是个有限的枚举：
bool, int, int8, int16, int32, int64,
uint, uint8, ..., float32, float64,
string, array, slice, map, struct,
ptr, func, interface, chan, ...
先看 Kind，再决定怎么处理——这是反射的基本套路。

### 反射的三大能力
1. 读取信息（最常用）
sq := &Square{Side: 5}
v := reflect.ValueOf(sq).Elem()   // Elem() 解一层指针，等价于 *sq
// 遍历结构体字段
for i := 0; i < v.NumField(); i++ {
    field := v.Type().Field(i)     // 字段的类型信息
    value := v.Field(i)            // 字段的值信息
    fmt.Printf("%s = %v (kind: %s)\n", field.Name, value, value.Kind())
}
// 输出: Side = 5 (kind: float32)


2. 修改值（需要指针）
sq := &Square{Side: 5}
v := reflect.ValueOf(sq).Elem()   // 必须传指针再 Elem，才能修改
sideField := v.FieldByName("Side")
sideField.SetFloat(100)            // 修改了 sq.Side
fmt.Println(sq.Side)               // 100
为什么必须传指针？ 因为 reflect.ValueOf(x) 是值拷贝。你改拷贝没有意义。传指针再 Elem() 才能定位到原数据——这和你写函数参数要传指针才能修改是同一个道理。
// ✗ 改不了
v := reflect.ValueOf(Square{Side: 5})
v.FieldByName("Side").SetFloat(100)  // panic! 不可寻址
// ✓ 能改
v := reflect.ValueOf(&Square{Side: 5}).Elem()
v.FieldByName("Side").SetFloat(100)  // ok

sq := &Square{Side: 5}
reflect.ValueOf(sq)              // 拿到 *Square 的 Value
         .Elem()                  // 等价于 *sq，拿到 Square 的 Value
         .FieldByName("Side")    // 找到 Side 字段
         .SetFloat(100)          // 修改，影响原数据
// 等价的普通代码：
(*sq).Side = 100


3. 调用方法
sq := &Square{Side: 5}
v := reflect.ValueOf(sq)
// 调用 Area 方法
result := v.MethodByName("Area").Call(nil)  // nil = 无参数
fmt.Println(result[0].Float())               // 25


## Go 中的面向对象

Go 有面向对象，但没有类和继承。它用三样东西替代了传统 OOP 的整套体系。

### struct + method = class

传统 OOP 把数据和方法绑在一起，Go 把它们分开写，再用 receiver 关联：

```go
type Dog struct {
    Name string
}

// (d Dog) 是 receiver，把 Speak 绑到 Dog 上
func (d Dog) Speak() {
    fmt.Println(d.Name + ": woof")
}

d := Dog{Name: "Rex"}
d.Speak()   // Rex: woof
```

方法可以随时、在任意文件里添加。数据定义一次，方法可以分散在各处。

### 嵌入 = 组合，不是继承

```go
type Animal struct {
    Name string
}

func (a Animal) Speak() string {
    return a.Name + " speaks"
}

// Dog "嵌入" Animal，不是"继承" Animal
type Dog struct {
    Animal          // 嵌入：Dog 包含了一个 Animal
    Breed string
}
```

用起来像继承：

```go
d := Dog{Animal: Animal{Name: "Rex"}, Breed: "Husky"}
fmt.Println(d.Name)     // Rex —— 可以直接访问嵌入字段的属性
fmt.Println(d.Speak())  // Rex speaks —— 可以直接调用嵌入字段的方法
```

但**本质是组合**：

```
继承：Dog 是一种 Animal        → Dog IS-A Animal
嵌入：Dog 里面有一个 Animal    → Dog HAS-A Animal
```

#### 方法覆盖是遮蔽，不是多态

```go
func (a Animal) Speak() string {
    return a.Name + " speaks"
}

func (d Dog) Speak() string {
    return d.Name + " barks"    // 遮蔽了 Animal 的 Speak
}

d := Dog{Animal: Animal{Name: "Rex"}, Breed: "Husky"}
d.Speak()               // "Rex barks" —— 用 Dog 的
d.Animal.Speak()        // "Rex speaks" —— 仍能调用被遮蔽的版本
```

传统继承中，`super.Speak()` 走的是虚函数表，和这里完全不同。

#### 多重嵌入

```go
type Dog struct {
    Animal          // 嵌入 Animal
    Pet             // 嵌入 Pet，传统单继承做不到
    Breed string
}
```

Go 的嵌入不会出现菱形继承问题——每个嵌入都是独立的实例。

### 接口隐式满足 = 最大的设计差异

```go
type Speaker interface {
    Speak() string
}

// Dog 完全不知道 Speaker 的存在
type Dog struct{ Name string }
func (d Dog) Speak() string { return d.Name + " barks" }

// 但 Dog 自动满足 Speaker，因为方法签名对上了
var s Speaker = Dog{Name: "Rex"}
fmt.Println(s.Speak())   // Rex barks
```

这叫**鸭子类型**：如果它走起来像鸭子、叫起来像鸭子，那它就是鸭子。不需要声明"我是鸭子"。

好处：你可以给第三方库的类型定义接口，不需要改它的代码。

### receiver 选值还是指针

```go
type Counter struct{ Count int }

// 值 receiver：操作副本，不影响原数据
func (c Counter) Increment1() {
    c.Count++          // 改的是副本
}

// 指针 receiver：操作原数据
func (c *Counter) Increment2() {
    c.Count++          // 改的是原数据
}

c := Counter{Count: 0}
c.Increment1()
fmt.Println(c.Count)   // 0，没变
c.Increment2()
fmt.Println(c.Count)   // 1，变了
```

选择规则：**要修改就用指针，不修改就用值。** 如果拿不准，统一用指针。

### 封装靠大小写，不靠 class

```go
type User struct {
    Name  string    // 大写：包外可见（public）
    email string    // 小写：包内可见（private）
}

func (u *User) SetEmail(e string) {   // 大写方法：public
    u.email = e                         // 包内可以访问 email
}
```

封装的单位是**包**，不是类。

### 没有构造函数，用工厂函数

```go
type User struct {
    Name  string
    email string    // 不想让外部直接设值
}

// 工厂函数代替构造函数
func NewUser(name, email string) *User {
    return &User{
        Name:  name,
        email: email,
    }
}

u := NewUser("Tom", "tom@example.com")
```

### 整体思维模型

```
传统 OOP 的世界：
  类 ──继承──▶ 类 ──implements──▶ 接口
  (数据+方法绑在一起，单向树状继承)

Go 的世界：
  struct ──嵌入──▶ struct    (组合，平面的)
  struct ──receiver──▶ method (数据和行为分离)
  struct ──隐式满足──▶ interface (鸭子类型，不需要声明)
```

Go 的设计哲学是**组合优于继承**。没有继承树，没有虚函数表，没有 super 调用。一切关系都是平面的：你包含什么，你就能用什么；你有什么方法，你就满足了什么接口。
